# TODO — 选股系统改善项目

投资框架（资产配置→宏观→行业→公司→估值→技术→仓位→卖出→复盘）と現行システムを
突き合わせて洗い出した改善点。優先度＝「目標(老後資金・最大回撤-10〜15%)への価値 × 実現性」。

現状の立ち位置：技術モメンタム・エンジンとしては検証済み（未知期間 期待値+2.49%/取引、
損益係数1.61）。ただしフレームワークの「攻撃10%」部分に相当し、估值・組合層リスクが欠落。

---

## 🔴 P1 — 致命的な空洞を塞ぐ

### 1. 估值闸门（バリュエーション・ゲート）を追加
- **現状の問題**: scannerの100点に估值が0点（RS25/底20/突破15/業績25/高値10/材料5）。
  純モメンタムなので「貴い銘柄」を弾けない。フレームワークでは估值15点。
- **改善**: `position_trend_scanner.py` に簡易バリュエーションを硬性ゲート or 減点で追加。
  業種別閾値（消費=PER、SaaS=EV/Sales、銀行=PB、資源=EV/EBITDA）。
  PER等は moomoo スナップショット（PE_TTM/PB）か財務から取得。
- **対象**: `outputs/position_trend_scanner.py`, `outputs/moomoo_openapi_screener.py`（PER取得）
- **工数**: 半日　**なぜ重要**: 高値掴み＝回撤の主因。老後資金の目標に直結。

### 2. 組合層の回撤コントロール + Core-Satellite 配分
- **現状の問題**: 単笔層のみ制御（単株10%/主題40%/ATR1%）。**組合総回撤の測定・約束がゼロ**。
  目標 -10〜15% と脱連。システムは「全部攻撃仓」で、適正な小比率に収まっていない。
- **改善**:
  - Core-Satellite 枠を定義（Core 60-70%=ETF積立/手動・NISA、Satellite 20-30%=本システム、Cash 5-15%）。
    システムは Satellite 枠内だけで ALERT→仓位 を出す。
  - 組合総権益の回撤トラッカー（跌5%復盤 / 跌10%降風険 / 跌15%停加杠杆 の自動提示）。
- **対象**: `outputs/position_sizer.py` の上位レイヤー（新規 `portfolio_manager.py` 等）, README
- **工数**: 1日（半分は方針決め）　**なぜ重要**: 最大回撤目標そのもの。最優先で目標と整合。

## 🟡 P2 — 質と復盤

### 3. 財務質量の自動取得（営収単一→多指標）
- **現状の問題**: 営収増速のみ、しかも手動 `fundamentals_us.csv`。ROE/ROIC/毛利率/FCF/負債率なし。
- **改善**: moomoo 財務接口で ROE/利益率/負債を自動取得し、scannerの「業績」を
  フレームワークの「財務質量20点」に格上げ。
- **対象**: `outputs/moomoo_openapi_screener.py`, `outputs/position_trend_scanner.py`
- **工数**: 半日〜1日

### 4. 復盤システムの稼働 + 定性フィールド必須化
- **現状の問題**: `trade_journal.csv` はヘッダのみ。週/月/季の復盤が未稼働。
  商业模式・护城河・管理层（自動化不可）の記録場所がない。
- **改善**:
  - trade_journal に商业模式/护城河/管理层の**必填定性メモ欄**を追加。
  - 月次復盤スクリプト（持仓涨跌原因・胜率・亏损归因を自動集計）。
- **対象**: `outputs/trade_journal.csv`（スキーマ）, 新規 `review.py`
- **工数**: 半日

- **対象**: `outputs/multibagger_finder.py`, `outputs/moomoo_backtest_runner.py`
- **工数**: 半日

### 6. 管理层・护城河は自動化しない（方針メモ）
- **判断**: 定性判断のためコード化しない。**項目4の trade_journal 定性欄に手書きで残す**。
  ここに工数を割かないことを明記（過剰実装の回避）。

## 推奨着手順
1 → 2 → 4 → 3 → 5（6は1〜5に内包）。
まず **1(估值闸门)** から：検証済みエンジンに最大の穴埋め、半日、既存scannerに直結。

## 不变条件（改善時も守る）
- レンズ/估值/レジームは「買い許可・産業ベータ」層。買い判定(status)は技術スコアで決める原則を維持。
- 取れないKPIは捏造しない（手動CSV注入で橋渡し、未取得は空欄）。
- 回測は未知期間の期待値で判断。サンプル100笔未満は参考値。
